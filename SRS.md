## 1. Introduction

### 1.1 Purpose
This Software Requirements Specification (SRS) defines the complete functional and non-functional requirements for **DwellSync**, a cross-platform property management mobile application. It is intended for developers, testers, project managers, and stakeholders to ensure a shared understanding of the system's scope and behavior.

### 1.2 Scope
DwellSync is a digital platform that bridges the communication and financial gap between landlords and tenants. It enables landlords to create and manage bills (rent, electricity, water, gas), track payments, manage tenant profiles, and communicate via in-app messaging. Tenants can view bills, make payments through multiple mobile financial services (bKash, Nagad, Rocket, etc.), track payment history, and message their landlord.

### 1.3 Definitions
- **Landlord:** Property owner who manages rental units and tenants.
- **Tenant:** Individual renting a property from a landlord.
- **Invite Code:** A unique 6-character alphanumeric code generated by a landlord for tenant registration.
- **Bill:** A financial statement comprising rent and utility charges (electricity, water, gas) with a due date.
- **Payment:** A financial transaction from tenant to landlord against a bill.
- **JWT:** JSON Web Token used for stateless authentication.

- Flutter SDK Documentation (>=3.0.0)

## 2. System Description

### 2.1 System Context
DwellSync is a client-server architecture comprising:
- **Mobile Client:** A Flutter-based cross-platform application (iOS, Android, Web, Windows, macOS, Linux).
- **API Server:** A Node.js + Express REST API hosted on a remote server.
- **Database:** MongoDB for persistent data storage.
- **File Storage:** Server-local filesystem for profile photo uploads.

### 2.2 System Architecture
The system follows a 3-tier architecture:
- **Presentation Tier:** Flutter mobile app with Provider state management.
- **Application Tier:** Express.js REST API with JWT-based authentication middleware.
- **Data Tier:** MongoDB with Mongoose ODM.

### 2.3 Design Constraints
- Must support both light and dark themes.
- Must support Bangladeshi Taka (BDT) and local mobile payment methods (bKash, Nagad, Rocket).
- JWT tokens must expire after 30 days.
- All API responses must be JSON-formatted.
- Profile photo uploads must be handled via multipart/form-data.

## 5. User Roles

### 5.1 Landlord
- Creates and manages bills for tenants.
- Views tenant list and tenant details.
- Receives and tracks payments.
- Communicates with tenants via in-app messaging.
- Manages their profile and invite code.
- Has full CRUD access on bills they create.

### 5.2 Tenant
- Views their own bills and payment status.
- Makes payments (full, half, or custom amount).
- Views payment and bill history.
- Views their landlord's profile.
- Sends messages to their landlord.
- Accesses help and FAQ.

### 6.2 Basic Flow Example: Make Payment (UC-07)

2. Tenant taps "Pay Now" or selects a pending bill.
3. System displays payment screen with bill details and amount options.
4. Tenant selects amount option (Full/Half/Custom).
5. Tenant selects a payment method (bKash/Nagad/Rocket/Card/Bank Transfer).
6. System displays method-specific instructions.
7. Tenant completes payment externally and enters the transaction ID.
8. Tenant taps "Submit Payment."
9. System validates and records the payment.
10. System updates the bill status to "paid."
11. System confirms payment success to the tenant.

## 7. Data Model

### 7.1 Entity-Relationship Overview

```
User (Landlord) --1:N--> User (Tenant)  [via inviteCode]
User (Landlord) --1:N--> Bill          [created by landlord]
User (Tenant)   --1:N--> Bill           [billed to tenant]
Bill             --1:1--> Payment       [paid against bill]
User (Landlord) --1:N--> Payment       [receives payments]
User (Tenant)   --1:N--> Payment       [makes payments]
User (Sender)   --1:N--> Message       [sends messages]
User (Receiver) --1:N--> Message       [receives messages]
```

### 7.2 User Schema

| Field | Type | Constraints | Description |
|---|---|---|---|
| `name` | String | Required | Full name |
| `email` | String | Required, Unique, Lowercase | Email address |
| `phone` | String | Required | Phone number |
| `password` | String | Required, bcrypt-hashed | Login password |
| `role` | String | Enum: `["tenant", "landlord"]` | User role |
| `landlordId` | ObjectId | Ref: User, Optional | Tenant's linked landlord |
| `inviteCode` | String | 6-char alphanumeric, Optional | Landlord's registration code |
| `photoUrl` | String | Optional | Profile photo URL |
| `isVerified` | Boolean | Default: `false` | Email verification flag |
| `createdAt` | Date | Auto-generated | Account creation timestamp |

### 7.3 Bill Schema

| Field | Type | Constraints | Description |
|---|---|---|---|
| `tenantId` | ObjectId | Ref: User, Required | Tenant billed |
| `landlordId` | ObjectId | Ref: User, Required | Landlord who created bill |
| `rentAmount` | Number | Default: 0 | Rent charge |
| `electricityBill` | Number | Default: 0 | Electricity charge |
| `waterBill` | Number | Default: 0 | Water charge |
| `gasBill` | Number | Default: 0 | Gas charge |
| `totalAmount` | Number | Required, Auto-calculated | Sum of all charges |

| Field | Type | Constraints | Description |

## 8. API Requirements

### 8.1 General
- Base URL: `http://localhost:5000/api` (configurable via environment)
- Response Format: JSON
- Authentication: JWT Bearer token in `Authorization` header
- File Uploads: `multipart/form-data` for profile photos

### 8.2 Authentication Endpoints (`/api/auth`)

| Method | Endpoint | Auth | Description |
|---|---|---|---|
| POST | `/register` | No | Register new user (landlord or tenant with invite code) |
| POST | `/login` | No | Authenticate and receive JWT |
| GET | `/me` | Yes | Get current user profile |
| PUT | `/me` | Yes | Update profile (multipart for photo) |
| GET | `/tenants` | Yes (Landlord) | List landlord's tenants |
| GET | `/invite/:code` | No | Validate invite code, return landlord info |
| PUT | `/invite/regenerate` | Yes (Landlord) | Generate new invite code |
| GET | `/:userId` | Yes | Get user by ID |

### 8.3 Bill Endpoints (`/api/bills`)

| Method | Endpoint | Auth | Description |
|---|---|---|---|
| POST | `/` | Yes (Landlord) | Create a bill |
| GET | `/` | Yes | List bills (landlord: all; tenant: own) |
| GET | `/:billId` | Yes | Get single bill details |
| PUT | `/:billId` | Yes (Landlord) | Update bill |
| DELETE | `/:billId` | Yes (Landlord) | Delete bill |
| GET | `/tenant/:tenantId` | Yes (Landlord) | Get bills for a specific tenant |

### 8.4 Payment Endpoints (`/api/payments`)

| Method | Endpoint | Auth | Description |
|---|---|---|---|
| POST | `/` | Yes | Create payment (auto-updates bill to "paid") |
| GET | `/` | Yes | List payments (landlord: all; tenant: own) |
| GET | `/bill/:billId` | Yes | Get payments for a specific bill |

### 8.5 Message Endpoints (`/api/messages`)

| Method | Endpoint | Auth | Description |
|---|---|---|---|
| POST | `/` | Yes | Send a message |
| GET | `/` | Yes | List all messages for current user |

## 10. Security Requirements

### 10.1 Authentication
- Passwords must be hashed using bcryptjs with a minimum salt rounds of 10.
- JWT tokens must be signed with a secret key stored in environment variables.
- JWT tokens must expire after 30 days (configurable via `.env`).
- All protected routes must verify JWT via middleware before processing requests.

### 10.2 Authorization
- Role-based access control: landlord-only endpoints must reject tenant requests with 403.
- Tenant users must only access bills and payments linked to their account.
- Landlord users must only access bills and tenants linked to their account.
- Message access must be restricted to sender and receiver only.

### 10.3 Data Protection
- User passwords must never be returned in any API response.
- The invite code endpoint must only expose landlord name and photo (not other personal data).
- Profile photo uploads must validate file type (images only) and enforce maximum file size.

## 11. Constraints & Assumptions

### 11.1 Constraints
- The application must run on both iOS and Android platforms.
- The backend must run on Node.js v18+ with MongoDB.
- The frontend must use Flutter SDK >=3.0.0 with Dart >=3.0.0.
- State management must use the Provider pattern.
- No real-time WebSocket connections — messaging is request-based.

### 11.2 Assumptions
- Users have reliable internet connectivity for most operations.
- Landlords and tenants are in Bangladesh and use BDT currency.
- Each tenant is linked to exactly one landlord.
- A landlord may have multiple tenants.
- Payment confirmation is based on user-provided transaction IDs (no real payment gateway integration).
- Email verification is deferred (isVerified field exists but not enforced).
